这节课我们讲 Bun。不要把它只理解成 Node.js 的一个替代品，它真正的价值是把现代 JavaScript 开发里一堆分散工具，重新收进一个统一的运行时里面。

今天会按照教学节奏来走：先讲为什么现在值得学 Bun，再讲安装和基本命令，然后逐步看它怎么处理 TypeScript、包管理、环境变量、文件、HTTP 服务、SQLite、测试、Shell 脚本和单文件打包。

大家听的时候重点抓一个问题：以前我们需要装很多工具才能完成的事情，在 Bun 里是不是可以用更少的配置直接完成。这个问题会贯穿整节课。

01 · 背景

为什么现在是学习 Bun 的好时机？

AI 编程工具让“模型 → 代码 → 执行 → 反馈”的循环变得极高频。

运行时启动速度、依赖安装速度、单文件分发能力，都会直接影响 Agent 产品体验。

01Claude Code 场景

AI Agent 需要频繁执行代码、跑测试、安装依赖。

02二进制分发

Bun 可以把 JS/TS 项目编译成单个可执行文件。

03基础设施掌控

核心产品越依赖运行时，越需要控制底层稳定性和性能。

先从背景讲起。文档里提到 Bun 团队加入 Anthropic，这件事在开发者圈子里引起讨论，核心不只是“谁收购了谁”，而是一个趋势：AI 模型公司开始重视底层开发基础设施。

原因很直接。像 Claude Code 这样的 AI 编程工具，不是简单生成一段代码就结束。它会改文件、运行命令、装依赖、跑测试，再根据结果继续修改。这个循环越高频，运行时的启动速度和执行效率就越关键。

所以我们学习 Bun，不只是追新工具，而是在理解未来 AI 编程工作流里，为什么更轻、更快、更容易分发的 JS 运行时会变得重要。

02 · 认知模型

Bun 不是一个命令替换器，
它是一个全家桶运行时

Runtime

直接运行 JS / TS / JSX / TSX

Package Manager

兼容 npm 生态，安装更快

Bundler

构建、打包、编译成二进制

Toolbox

测试、Shell、SQLite、Web API

Node.js 时代ts-node / tsxdotenvJest / VitestWebpack / Vite多个工具拼装
Bun 时代bun一套工具链覆盖日常开发

这一页先建立认知模型。学习 Bun 不是把 npm install 换成 bun install 这么简单。如果只是记命令，过两天就忘了。

更准确地说，Bun 是一个 全家桶运行时。它既能运行代码，又能安装依赖，还能构建、测试、写脚本、处理 SQLite，甚至直接启动 HTTP 服务。

这和 Node.js 时代的体验差异很大。以前我们写一个 TS 项目，可能要配 ts-node 或 tsx；读环境变量要 dotenv；测试要 Jest 或 Vitest；打包还要 Webpack、Rollup 或 Vite。Bun 的目标是把这些高频能力都收进一个工具里。

03 · 安装

安装 Bun：先确认命令能跑起来

macOS / Linux / WSL

curl -fsSL https://bun.sh/install | bash

Windows PowerShell

powershell -c "irm bun.sh/install.ps1 | iex"

通过 npm 安装

npm install -g bun

验证版本

bun --version

安装这一页很简单，但课堂上一定要让大家真的跑一遍。不同系统命令不一样：macOS、Linux 和 WSL 通常用 curl 脚本；Windows 可以直接用 PowerShell；如果已经有 Node 环境，也可以通过 npm 全局安装。

安装完成以后，不要急着写代码，先运行 bun --version。只要能看到版本号，就说明环境变量和执行路径基本没问题。

这里顺便强调一下：学习新工具时，第一步不是理解所有高级特性，而是把最小反馈链路跑通。能安装、能查看版本、能执行一个文件，后面所有内容才有基础。

04 · 直接运行 TypeScript

彻底告别“先配转译器”

Node.js 常见链路

  • 运行 TS：装 ts-node 或 tsx
  • 运行 JSX：配置 Babel 或构建工具
  • 预览项目：先理解一堆配置

Bun 链路

bun index.ts
bun component.tsx
bun --hot server.ts
提示：tsconfig.json 仍然有用，但更多用于类型检查和编辑器体验。

第一个真正影响开发体验的能力，是 Bun 可以直接运行 TypeScript 和 TSX。传统 Node.js 环境里，运行 TS 往往要装 ts-node 或 tsx，运行 JSX 还可能需要 Babel 或构建工具。

Bun 的做法更直接：你写什么文件，就直接 bun 这个文件。比如 bun index.ts，或者 bun component.tsx。这对教学和小实验特别友好，因为学生不需要先被工具链劝退。

不过也要讲清楚，Bun 不是让 tsconfig.json 完全消失。它仍然会影响类型检查、路径别名和编辑器行为。只是运行代码这一步，不再依赖复杂转译配置。

05 · 包管理

极速包管理：兼容 npm 生态，减少等待

安装全部依赖bun install
添加运行依赖bun add axios
添加开发依赖bun add -d @types/bun
运行脚本bun run dev
关键点

Bun 兼容 node_modules 结构，同时使用自己的锁文件来提升读取和安装速度。

包管理这一部分很适合现场演示。因为它不是抽象概念，大家能直接感受到速度差异。常用命令也不难记，bun install 对应 npm install，bun add 对应安装依赖。

这里要强调两个点。第一，Bun 不是重新发明整个生态，它仍然兼容 npm 包和 node_modules 结构，所以迁移成本没有想象中那么高。

第二，速度对日常开发很重要。尤其在 AI Agent 场景里，安装依赖可能不是人偶尔点一次，而是自动化流程频繁触发。每次少等几秒，整体反馈循环就会明显变快。

06 · CLI 工具箱

很多第三方库，在 Bun 里变成了内置能力

ENV

环境变量

默认读取 .env，直接用 process.env 或 Bun.env。

FILE

文件操作

Bun.file() 返回懒加载文件对象，写入用 Bun.write()。

WEB

Web API

原生支持 fetch、Request、Response、WebSocket。

# .env
PORT=3000
API_KEY=fengfeng_666

// index.ts
console.log(process.env.PORT);
console.log(Bun.env.API_KEY);

这页讲 Bun 的工具箱思路。很多 Node.js 里要靠第三方库补齐的能力，在 Bun 里是默认内置的。

第一个例子是环境变量。以前我们常常装 dotenv，然后在入口文件顶部写 require('dotenv').config()。Bun 默认就会处理 .env，所以可以直接读 process.env，也可以用 Bun.env。

第二个例子是文件操作。Bun.file() 的体验更接近 Web 标准里的 Blob，对读取文本、二进制、写文件都更简洁。第三个是 Web API，前端熟悉的 fetch、Request、Response 在服务端也能自然使用。

07 · 文件 API

Bun.file：用更少代码处理文件和流

const file = Bun.file("input.txt");

const text = await file.text();
const arrayBuffer = await file.arrayBuffer();

await Bun.write("output.txt", "这是 Teacher Fengfeng 写入的新内容");

const response = await fetch("https://example.com");
await Bun.write("website.html", response);

export {}; // 让编辑器把文件视为 ESM，避免顶层 await 报红
懒加载Blob 风格 API支持把 Response 写成文件

文件 API 可以作为课堂练习。先准备一个 input.txt，然后用 Bun.file() 读取文本，再用 Bun.write() 写一个新文件。

这里有两个教学点。第一，Bun.file() 返回的不是马上把文件全部读进内存的字符串，而是一个文件对象，真正读取发生在调用 text() 或 arrayBuffer() 的时候。

第二，Bun 很多 API 都贴近 Web 标准。比如从 fetch 得到的 Response，可以直接写进文件。这个写法对做下载器、缓存器、代理工具都很顺手。

如果编辑器里顶层 await 报红，可以在文件末尾加 export {}，让 TypeScript 把当前文件识别成 ESM。

08 · HTTP 服务

Bun.serve：内置高性能 Web 服务器

Bun.serve({
  port: 3000,
  fetch(req) {
    return new Response("Welcome to Bun!");
  },
});

// 热重载
bun --hot index.ts
HOT热重载

修改代码后服务快速更新，不需要手动重启。

TLSHTTPS 配置

通过 tls 字段传入 key、cert、ca。

LIMIT安全参数

配置 hostname、maxRequestBodySize、reusePort。

接下来是 Bun 的杀手锏之一：Bun.serve。它内置了一个高性能 Web 服务器，所以写最小服务非常简单。

左边这段代码就是完整服务。监听 3000 端口，请求进来以后返回一个 Response。这个 Response 和浏览器 Fetch API 里的 Response 是同一套概念，所以前后端知识可以复用。

教学时可以先跑 bun index.ts，再改成 bun --hot index.ts。让学生修改返回文本，刷新浏览器看到变化，就能理解热重载的价值。

实际项目里还要关注安全和部署参数，比如限制上传体积、绑定 localhost、HTTPS 证书、Linux 下的 reusePort，这些都是从玩具项目走向生产服务时必须知道的。

09 · 路由

两种路由思路：文件路由 or 框架路由

内置 FileSystemRouter

const router = new Bun.FileSystemRouter({
  style: "nextjs",
  dir: "./pages",
});

Bun.serve({
  fetch(req) {
    const match = router.match(req);
    if (match) {
      return import(match.filePath)
        .then(mod => mod.default(req, match));
    }
    return new Response("404", { status: 404 });
  },
});

Elysia 风格

import { Elysia } from "elysia";

new Elysia()
  .get("/", () => "首页")
  .get("/user/:id", ({ params: { id } }) => `用户ID: ${id}`)
  .post("/upload", ({ body }) => ({ status: "success" }))
  .listen(3000);

讲完最小 HTTP 服务，再讲路由。这里可以给学生两个选择：如果喜欢 Next.js 那种目录即路由的思路，可以看 Bun 内置的 FileSystemRouter；如果喜欢 Express 风格，就用 Elysia 这样的框架。

左边的例子里，style: "nextjs" 表示支持类似 [id].ts 的动态路由语法。请求进来以后先匹配文件，再动态 import 对应模块。

右边是 Elysia，更像我们熟悉的后端路由写法。.get、.post 链式注册路由，路径参数也比较直观。

课堂上建议不要一开始就上框架。先用 Bun.serve 理解底层，再引入 Elysia，会更清楚框架到底帮我们封装了什么。

10 · 数据、测试、脚本

Bun 把后端常用小工具也放进来了

SQLite

import { Database } from "bun:sqlite";

const db = new Database(":memory:");
const query = db.query("SELECT 'Hello Bun!' as message");
console.log(query.get());

Test

import { expect, test } from "bun:test";

test("sum", () => {
  expect(1 + 1).toBe(2);
});

bun test

Shell

import { $ } from "bun";

await $`mkdir -p new_folder`;
const files = await $`ls`.text();
console.log(files);

这一页把三个能力放在一起讲：SQLite、测试框架和 Bun Shell。它们共同体现了 Bun 的设计取向：把高频、基础、容易重复安装的能力内置进来。

SQLite 很适合教学和小型工具，因为不需要单独启动数据库服务。Bun 内置 bun:sqlite，不需要安装驱动，也不需要编译 C++ 插件。

测试方面，bun test 的语法兼容 Jest 风格，test、expect、describe 这些概念学生已经很熟悉。

Bun Shell 则解决跨平台脚本问题。以前在 Node.js 里写脚本，经常要处理 Windows 和 Linux 命令差异，或者引入 zx。Bun 直接提供 $ 模板字符串，写工具脚本会轻很多。

11 · 分发与进阶

单文件打包 + 宏：从工具到产品

单文件二进制

bun build ./index.ts --compile --outfile my-app

把 Bun 运行时、你的代码、依赖一起打进可执行文件。

编译时宏

import { getVersion } from "./utils.ts" with { type: "macro" };

console.log(getVersion());
// 编译后直接变成 "1.0.0"
适合场景

CLI 工具、AI Agent、内部运维工具、桌面辅助脚本、无需用户预装 Node/Bun 的分发包。

这一页讲两个更进阶的能力。第一个是单文件打包。命令是 bun build ./index.ts --compile --outfile my-app，它会把运行时、代码和依赖一起打进一个可执行文件。

这对分发 CLI 工具非常有价值。Node.js 时代，用户通常要先装 Node，再全局安装包，或者我们要借助比较重的第三方打包方案。Bun 的 compile 是原生能力，交付体验更直接。

第二个是宏，也就是编译时执行代码。比如打包时读取 package.json 的版本号，然后把结果直接硬编码进输出文件。这个能力适合处理编译期可确定的数据。

这里不用让初学者一上来就写宏，但要让他们知道 Bun 不只是快，它还在尝试改变 JS 工具的分发方式。

13 · 总结

今天需要带走的 5 个判断

01Bun 的核心价值是减少工具链碎片runtime
02TS/JSX、.env、fetch、WebSocket 都可以更自然地跑DX
03Bun.serve 适合快速做 HTTP 服务和课堂实验server
04SQLite、test、Shell 让小工具开发更轻toolbox
05--compile 让 JS/TS 工具具备更好的分发形态ship

课堂练习：用 Bun 写一个带路由、文件读写、测试和单文件打包的小 CLI/API。

#Bun #JavaScript #Teaching

最后总结一下。今天不是为了让大家背下所有 Bun API，而是建立五个判断。

第一，Bun 的核心价值是减少工具链碎片；第二，TypeScript、环境变量、Web API 这些日常能力可以更自然地跑起来；第三，Bun.serve 适合快速做 HTTP 服务和教学实验；第四，SQLite、测试、Shell 让小工具开发更轻；第五，--compile 让 JS 和 TS 工具有了更好的分发形态。

课后练习建议做一个小项目：用 Bun 写一个带路由的 API，读取一个本地文件，写一个测试，最后编译成单文件。把这条链路跑通，就算真正入门了。
